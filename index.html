Testing

¿Por qué Escribir Tests?

1. Confianza: Garantiza que tu código hace lo que esperas.
2. Seguridad: Permite refactorizar y añadir nuevas características sin romper lo que ya funcionaba (evita regresiones).
3. Documentación: Los tests son ejemplos vivos de cómo debe usarse tu código.

2 / 20

Tipos de Tests (Conceptos)

Unit Test (Prueba Unitaria): Prueba una pequeña pieza de código (una función, un método) de forma aislada.

Integration Test (Prueba de Integración): Prueba cómo interactúan varias piezas de código juntas.

End-to-End Test (Prueba de Extremo a Extremo): Prueba el flujo completo de la aplicación, como lo haría un usuario.

El Clásico: unittest

El framework de testing integrado en la librería estándar de Python. Está basado en clases y es muy completo.

4 / 20

Estructura de unittest

Se crea una clase que hereda de `unittest.TestCase` y los métodos de prueba deben empezar con `test_`.

import unittest

class MiTest(unittest.TestCase):
    def test_algo(self):
        # ... aquí va el código de la prueba ...
        pass
5 / 20

Aserciones en unittest

Se usan métodos especiales como `self.assertEqual()` para verificar que el resultado es el esperado.

def test_suma(self):
    resultado = 2 + 2
    self.assertEqual(resultado, 4)
    self.assertTrue(resultado > 0)

Ejecutar unittest

Se ejecuta desde la terminal. El comando buscará y ejecutará todos los tests.

python -m unittest discover

El Moderno: pytest

El framework de testing estándar de facto en la comunidad. Es más simple, más legible y mucho más potente que `unittest`.

8 / 20

¿Por qué pytest es Mejor?

1. Menos código: No requiere clases.
2. Aserciones simples: Usa la palabra clave `assert` normal de Python.
3. Fixtures: Un sistema increíble para gestionar el estado de los tests.
4. Ecosistema enorme: Cientos de plugins para todo tipo de testing.

9 / 20

Ejemplo con pytest

Los tests son simplemente funciones que empiezan con `test_` en archivos que empiezan con `test_`.

# En calculadora/operaciones.py
def sumar(a, b):
    return a + b

# En tests/test_operaciones.py
from calculadora import operaciones

def test_sumar():
    assert operaciones.sumar(2, 3) == 5
    assert operaciones.sumar(-1, 1) == 0

Fixtures en pytest

Una fixture es una función que prepara datos o recursos para un test. Se usa para evitar repetir código de configuración.

import pytest

@pytest.fixture
def lista_de_numeros():
    """Devuelve una lista simple para usar en los tests."""
    return [1, 2, 3, 4, 5]

def test_suma(lista_de_numeros):
    assert sum(lista_de_numeros) == 15

Parametrización

Permite ejecutar el mismo test con múltiples combinaciones de datos, sin repetir código.

@pytest.mark.parametrize("a, b, esperado", [
    (1, 2, 3),
    (-1, 1, 0),
    (10, -5, 5)
])
def test_sumar_parametrizado(a, b, esperado):
    assert sumar(a, b) == esperado

TDD: Test-Driven Development

Una metodología de desarrollo que sigue 3 pasos:

1. Rojo: Escribe un test que falla porque la funcionalidad aún no existe.
2. Verde: Escribe el código más simple posible para que el test pase.
3. Refactor: Limpia y mejora el código, asegurándote de que el test siga pasando.

Code Coverage

Mide qué porcentaje de tu código es ejecutado por tus tests. Ayuda a encontrar partes de tu aplicación que no están siendo probadas.

pip install pytest-cov

# Ejecutar tests y generar reporte de cobertura
pytest --cov=mi_proyecto

Mocking (Simulación)

La técnica de reemplazar partes de tu sistema (como una API externa o una base de datos) con objetos falsos para poder probar tu código de forma aislada.

16 / 20

Ejemplo de Mocking

Usamos `unittest.mock` para simular una petición a una API y evitar que el test dependa de una conexión a internet.

from unittest.mock import patch

@patch('requests.get')
def test_mi_funcion(mock_get):
    # Configuramos el mock para que devuelva un resultado falso
    mock_get.return_value.json.return_value = {"nombre": "Ana"}
    
    # ... ahora llamamos a nuestra función que usa requests.get ...
    resultado = mi_funcion_que_usa_api()
    assert resultado == "Ana"

Estructura de un Proyecto

Normalmente, los tests viven en una carpeta separada llamada `tests/` en la raíz de tu proyecto.

/mi_proyecto
|-- venv/
|-- /mi_proyecto
|   |-- __init__.py
|   |-- mi_codigo.py
|-- /tests
    |-- test_mi_codigo.py

Regla de Oro

Un test que no se ejecuta automáticamente de forma regular es un test que no sirve de nada. Integra el testing en tu flujo de trabajo diario.

Conclusión

Escribir tests es una inversión que paga dividendos en estabilidad, mantenibilidad y paz mental. pytest es la herramienta moderna y recomendada para hacer de esta práctica algo simple y poderoso.
